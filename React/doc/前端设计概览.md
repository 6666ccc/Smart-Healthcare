## 4. 三层门户体系

这是系统的**核心设计模式**。同一套代码，根据用户角色呈现完全不同的功能界面。

### 4.1 三种门户

| 门户标识 | 中文名称 | 典型用户 | 角色判定依据 |
|----------|----------|---------|-------------|
| `admin` | 管理端 | admin | `portalType` / 默认兜底 |
| `doctor` | 医生端 | doctor01 | `portalType` / 用户名正则 `^doctor\d*$` |
| `patient` | 患者端 | patient01 | `portalType` / 用户名 `patient*` 前缀 |

### 4.2 角色判定逻辑 (`views/Home/role.js`)

优先级（从高到低）：
1. `user.portalType` — 后端 LoginVO 显式返回
2. `user.userType` / `user.portal` — 备选字段
3. `user.roleCode` / `user.roleName` / `user.username` — 前端本地兜底规则

未匹配到任何规则时默认返回 `admin`。

### 4.3 各门户功能差异

#### PC 端侧边栏导航

| 收费管理 | 检查报告 | 门诊缴费 |

## 5. 响应式双端设计

### 5.1 响应式策略

所有业务页面遵循统一的 **PC / Mobile 分离** 模式：

```
View/index.jsx          ← 入口组件
  │
  ├── useIsPc()          ← 监听 matchMedia('(min-width: 1024px)')
  │
  ├── isPc === true  →  View/pc/index.jsx    ← 桌面端布局
  └── isPc === false →  View/mobile/index.jsx ← 移动端布局
```

断点值：**1024px**。`>= 1024px` 判定为 PC，`< 1024px` 判定为移动端。

### 5.2 PC 端布局

通过 `PcLayout` 组件实现统一的三栏式后台布局：

```

│ Sidebar  │  ├──────────────────────────────────┤│
│          │  │                                  ││
│  Logo    │  │  Greeting（问候语 + 日期）         ││
│  Nav     │  │                                  ││
│  ...     │  │  Page Content                    ││
│          │  │                                  ││
│  AI 按钮 │  ├──────────────────────────────────┤│
│          │  │  Footer                          ││
```

## 6. 路由系统

### 6.1 路由表

```js
// src/router/index.jsx — createBrowserRouter 配置
[
  // 未登录用户可访问
  { element: <GuestOnly />, children: [
    { path: '/',       element: <Login /> },
    { path: '/login',  element: <Login /> },
  ]},
  
  // 需登录
  { element: <RequireAuth />, children: [
    { path: '/home',                    element: <Home /> },
    { path: '/user',                    element: <User /> },
    { path: '/assistant',               element: <Assistant /> },
    { path: '/registration',            element: <Registration /> },
    { path: '/registration/:id',        element: <RegistrationDetail /> },
    { path: '/department',              element: <Department /> },
    { path: '/payment',                 element: <Payment /> },
    { path: '/payment/:id',             element: <PaymentPay /> },
    { path: '/consultation',            element: <Consultation /> },
    { path: '/dispense',                element: <Dispense /> },
    { path: '/patients',                element: <PatientList /> },
    { path: '/schedules',               element: <ScheduleManage /> },
    { path: '/drugs',                   element: <DrugManage /> },
  ]},
  
  // 兜底
  { path: '*', element: <Navigate to="/login" replace /> },
]
```

### 6.2 路由守卫

```js
// RequireAuth — 未登录 → /login
function RequireAuth() {
  const { isAuthenticated } = useAuth()
  if (!isAuthenticated) return <Navigate to="/login" replace />
  return <Outlet />
}

// GuestOnly — 已登录 → /home
function GuestOnly() {
  const { isAuthenticated } = useAuth()
  if (isAuthenticated) return <Navigate to="/home" replace />
  return <Outlet />
}
```

## 7. 状态管理 & 认证流

### 7.1 AuthProvider 结构

```
AuthProvider (store/auth.jsx)
│
├── user                 ← localStorage key: "wenrun_user" (JSON)
├── token                ← localStorage key: "wenrun_token" (string)
├── loading              ← 请求进行中标识
├── isAuthenticated      ← Boolean(token)
│
├── login(username, password)
│     → POST /api/auth/login (skipAuth: true)
│     → 保存 token + user 到 localStorage
│     → 更新 state
│
└── logout()
      → POST /api/auth/logout
      → 清除 token + user（即使网络异常也清理本地态）
      → 重置 state
```

### 7.2 Token 传递

所有 API 请求通过 Axios **请求拦截器** 自动注入：

```js
config.headers.Authorization = `Bearer ${token}`
config.headers['X-Token'] = token
```

例外路径（跳过认证）：`/api/health`、`/api/auth/login`。

### 7.3 响应处理

响应拦截器自动提取业务数据：

- `response.data.code === 200` → 返回 `response.data.data`
- 其他 code → `Promise.reject(Error(message))`
- 网络异常 → `Promise.reject(Error(错误信息))`

## 9. 核心业务模块

### 9.1 登录 (`views/Login`)

- 纯表单页面，不区分 PC/Mobile
- 内联 SVG 图标组件（User、Lock、Logo）
- 演示账号提示：`admin` / `doctor01` / `patient01`，密码均为 `password`
- 登录成功后 `navigate('/home', { replace: true })`

### 9.2 首页 (`views/Home`)

这是最复杂的模块，承载了门户分流的全部逻辑：

```
Home/index.jsx → useIsPc() 判断
  ├── Home/pc/index.jsx  → getPortalType(user)
  │     ├── AdminHomePc    ← useHomeData(PORTAL.ADMIN)
  │     ├── DoctorHomePc   ← useHomeData(PORTAL.DOCTOR)
  │     └── PatientHomePc  ← useHomeData(PORTAL.PATIENT)
  │
  └── Home/mobile/index.jsx → getPortalType(user)
        ├── AdminHomeMobile
        ├── DoctorHomeMobile
        └── PatientHomeMobile
```

**PC 端首页布局**（以管理端为例）：
1. 统计卡片行（今日挂号 / 接诊 / 收费 / 待办）
2. 快捷功能卡片（挂号 / 接诊 / 收费 / 发药）
3. 底部双栏：
   - 左：待办任务（待诊患者 / 待收费 / 待发药）
   - 右：库存预警 + 运营提示 + 医院公告

**数据加载**采用 `Promise.allSettled` 并行请求，单个接口失败不影响其他数据展示。

### 9.3 挂号管理 (`views/Registration`)

- **列表页**：全部 / 待诊 Tab 切换，展示患者名、科室、日期、状态

- **挂号详情** (`/registration/:id`)：独立页面（非双端适配），展示详情 + 取消操作

### 9.4 接诊管理 (`views/Consultation`)

## 14. 已知问题与改进建议

### 14.1 `useIsPc` 代码重复

**现状**：Home、Registration、Consultation、Department、User、Assistant、Payment 共 7 个页面各自内联了相同逻辑的 `useIsPc`（约 12 行/处），而 Dispense、PatientList、ScheduleManage、DrugManage 正确使用了共享 Hook。

**建议**：统一替换为 `import { useIsPc } from '../../hooks'`。

### 14.2 PC 端侧边栏重复

**现状**：`Registration/pc/index.jsx` 和 `Consultation/pc/index.jsx` 各自渲染了完整的侧边栏（sidebar + header + footer 约 70 行），而非复用 `Home/pc/PcLayout.jsx`。

**建议**：将业务页面改造为使用 `PcLayout` 的 children 模式。

### 14.3 工具函数重复

`utils/index.js` 与 `views/Home/utils.js` 中存在相同函数：
- `formatTime()` — 两边各自实现
- `formatMoney()` — 两边各自实现

**建议**：统一到 `utils/index.js`，Home 直接引用。

### 14.4 无 TypeScript

全项目使用 `.jsx`/`.js`，无类型约束。API 响应结构、状态映射、组件 Props 均无类型定义。

**建议**：渐进式迁移，优先给 API 模块和共享组件添加类型。

### 14.5 无数据缓存层

每个页面自行管理 `useState` + `useCallback` + `useEffect` 数据加载模式，无请求去重、缓存、自动重新验证机制。

**建议**：考虑引入 `@tanstack/react-query` 或 `swr` 以改善数据管理。

### 14.6 演示账号硬编码

登录页底部的演示账号提示（"演示：admin / doctor01 / patient01，密码均为 password"）直接硬编码在 JSX 中，不便于动态配置。

### 14.7 Registration Detail 非双端适配

`Registration/Detail.jsx` 和 `Payment/Pay.jsx` 是两个独立详情页面，没有做 PC/Mobile 分离，在移动端可能体验不佳。
