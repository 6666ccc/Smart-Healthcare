# 患者端双模式 Agent 互联网医院设计

## 1. 目标

将现有前端重构为仅面向患者的双模式互联网医院：

- 新版采用 Agent-first 的“随身诊室”，患者通过对话表达需求，由 Agent 编排查询、分诊、挂号与缴费等流程。
- 传统版保留现有菜单式患者自助服务，降低既有用户的迁移成本。
- 登录后由患者主动选择使用方式，两种模式共享账户和业务数据，并可随时切换。
- 删除医生端的路由、布局和页面，不改动后端医生业务接口。

成功标准是：患者进入新版后首先感受到的是一个理解就医需求的智能体，而不是在传统功能菜单上额外增加了聊天入口。

## 2. 范围与非目标

### 范围内

- 患者登录后的模式选择页。
- 新版 Agent 工作台的手机优先重构与桌面渐进增强。
- 传统患者首页、挂号、科室、缴费和个人中心的继续可用。
- 新版与传统版之间的双向切换。
- 医生端前端代码与路由的移除。
- 对话、患者上下文、结构化结果卡和任务面板的状态设计。
- 响应式、可访问性、错误状态和医疗安全提示。

### 非目标

- 不修改 Java 或 Python 后端的医生业务能力。
- 不新增处方、药品、库存或其他医护管理能力。
- 不让 AI 替代医生进行诊断或开具处方。
- 不在本轮加入语音输入、视频问诊、推送通知或新的支付渠道。
- 不迁移 TypeScript，不引入新的全局状态或 UI 框架。

## 3. 产品结构与路由

### 3.1 登录与模式选择

患者登录成功后统一进入 `/mode-select`。页面提供两张明确的体验卡：

1. “新版 · AI 随身诊室”，进入 `/assistant`。
2. “传统版 · 自助服务”，进入 `/home`。

本地保存键 `wenrun_patient_mode`，值为 `agent` 或 `classic`。每次重新登录仍显示模式选择页，只用上次选择高亮推荐项，不自动跳过选择。模式切换不退出登录，也不清空患者上下文。

### 3.2 新版路由

- `/assistant`：Agent 工作台主入口。
- 复杂业务不增加新的顶级导航；挂号和缴费以工作台内的任务面板承接。
- 当浏览器直接访问 `/assistant` 时，仍由现有登录与患者权限守卫保护。

### 3.3 传统版路由

- `/home`：传统患者首页。
- `/registration` 与 `/registration/:id`：挂号列表和详情。
- `/department`：科室查询。
- `/payment` 与 `/payment/:id`：账单列表和支付。
- `/user`：患者个人中心。

传统版沿用现有底部导航和桌面侧栏，但增加“体验 AI 新版”入口。新版顶部提供“切换传统版”入口。

### 3.4 医生端删除边界

移除所有 `/doctor/*` 路由、医生端页面和医生端专用布局。登录后的前端只接受患者门户；医生账号不再进入任何前端工作台，而是看到不支持该账号类型的明确提示。后端接口和数据库不在此次删除范围内。

## 4. 推荐体验：“随身诊室”

### 4.1 设计原则

- **对话是主线**：欢迎语、消息流和输入器占据视觉中心，功能入口不组成传统宫格首页。
- **任务不跳场**：复杂操作在底部任务面板中完成，关闭面板后回到原对话位置。
- **上下文适时出现**：下一次预约、待缴费用和最近就诊只在相关时机展示，不长期挤占主屏。
- **每一步可确认**：涉及时间、医生、金额和业务提交时，显示结构化摘要并要求患者确认。
- **手机体验优先**：关键操作位于拇指热区，桌面端只展开信息，不创造另一套操作模型。

### 4.2 手机布局

从上到下分为五层：

1. 顶栏：品牌标识、Agent 在线状态、模式切换。
2. 欢迎态或对话线程：首次进入显示 Agent 护理灯、问候和简短能力说明；开始对话后切换为消息流。
3. 轻量上下文：建议问题、下一待办和结构化结果卡。
4. 固定输入器：文字输入、发送状态和停止生成操作。
5. 任务面板：从底部抽屉展开挂号、缴费等分步流程。

任务面板展开时保留对话背景和当前任务标题；提交成功后面板关闭，并在消息流中插入一张可追溯的完成卡。

### 4.3 桌面布局

- 中间：与手机一致的对话主线和任务流程。
- 左侧：会话历史、新建对话和患者账号摘要。
- 右侧：下一次预约、待缴费、最近就诊和当前任务进度。
- 在中等宽度隐藏右栏，在窄屏折叠为手机单列。

桌面端不恢复传统菜单侧栏；传统服务必须通过模式切换进入 `/home`。

## 5. 核心界面与组件边界

### 5.1 模式选择页

- `ModeSelectPage` 只负责展示两种体验、读取上次选择和完成路由跳转。
- `ModeCard` 负责单张模式卡的视觉、说明和选中态。
- 页面文案使用“今天，想怎样使用医院？”，避免技术化的“系统版本选择”。

### 5.2 Agent 工作台

- `AssistantShell`：响应式外壳和区域编排。
- `AssistantHeader`：品牌、在线状态、模式切换和移动端历史入口。
- `ConversationThread`：欢迎态、消息列表、流式状态和滚动定位。
- `ChatComposer`：输入、发送、停止和医疗免责声明。
- `ContextSummary`：预约、费用和最近就诊的独立摘要状态。
- `TaskSheet`：移动端底部抽屉、桌面端右侧任务面板的共同容器。
- `ActionCard`：对话中的查询结果、确认请求和完成结果。
- `ConversationHistory`：本地会话的新建、切换和删除。

业务数据与 UI 状态分离：对话会话、患者上下文和当前任务分别由独立 Hook 管理，避免继续扩张现有单文件 `Assistant/index.jsx`。

### 5.3 任务类型

首轮只实现三类前端呈现：

1. **查询卡**：预约、待缴费和就诊记录，仅展示数据并提供下一步。
2. **挂号任务**：科室与时间、医生、最终确认三个步骤。
3. **缴费任务**：账单明细、支付方式、金额确认三个步骤。

未知 Agent 动作不动态生成任意表单，而是显示安全提示，并提供传统版或人工服务的降级入口。

## 6. 数据流与状态

### 6.1 页面初始化

进入 `/assistant` 后并行读取患者挂号、待缴费和就诊记录。各请求相互隔离：一个请求失败只影响对应摘要卡，不阻塞对话输入。加载期间使用稳定骨架，不使用整页遮罩。

### 6.2 对话流

1. 患者提交文本后立即写入当前会话。
2. 调用现有 `chatStream`，用同一 `conversationId` 接收流式内容。
3. 流式文本持续更新最后一条 Agent 消息。
4. 如果响应映射到支持的任务类型，消息后渲染对应 `ActionCard`。
5. 患者打开任务面板并完成必要确认。
6. 业务 API 成功后刷新受影响的上下文，并将完成卡写回对话。

前端只消费白名单任务类型；未知类型以普通文本或安全降级卡显示。

### 6.3 本地状态

- 会话历史继续使用 `localStorage`，但需要版本化存储结构并容忍旧数据损坏。
- 模式偏好单独保存，不与会话内容混合。
- 不在本地保存完整身份证号、支付凭据或医疗附件。
- 当前打开的任务不跨页面持久化，刷新后由患者重新确认，避免误提交。

## 7. 视觉系统

### 7.1 风格

整体方向为安静、有生命感的“随身诊室”，避免蓝白后台、紫色渐变 AI 工具和大面积玻璃卡片。

- 深海青 `#102E3B`：品牌、主文字和用户消息。
- 护理绿 `#137D75`：Agent 状态、成功和选中。
- 暖米白 `#F6F2E8`：主画布。
- 琥珀 `#DFA548`：主要确认动作。
- 危险红仅用于急症、错误和不可逆操作，不作为装饰色。

标题使用有温度的中文宋体或衬线字体栈，正文使用高可读性的中文无衬线字体栈。字体加载失败时必须有本地回退，不依赖外部字体才能完成首屏。

### 7.2 动效

- Agent 护理灯使用低频呼吸光，表达在线而不是“正在诊断”。
- 欢迎内容按品牌、标题、建议问题和输入器顺序在 600ms 内完成入场。
- 消息渐入，任务面板使用短距离上移；不使用连续漂浮或高频闪烁。
- `prefers-reduced-motion: reduce` 时关闭非必要动画和顺滑滚动。

### 7.3 响应式与触控

- 设计基准宽度为 390px，并覆盖 360px 至桌面宽度。
- 可触控按钮最小高度 44px，主要动作建议 48px。
- 输入器考虑移动浏览器安全区和虚拟键盘，不被固定底栏遮挡。
- 任何宽度不得出现页面级横向滚动。

## 8. 异常、空状态与医疗安全

### 8.1 对话异常

- 首个 token 前显示安静的“正在整理信息”状态。
- 流式请求失败时保留患者消息，显示“重新发送”和“切换传统版”，不复制失败的 Agent 内容。
- 患者可停止生成；停止后保留已收到文本并标记“回答已停止”。
- 离线时禁用发送并提示检查网络，恢复连接后无需刷新页面。

### 8.2 上下文异常

- 无预约、无费用、无就诊记录分别使用具体空状态，不显示 `0` 值仪表盘。
- 单个接口失败时只在对应卡片显示重试按钮。
- 任务提交失败保留患者已选内容，允许重试或返回上一步。
- 提交按钮在请求中锁定，防止重复挂号和重复支付。

### 8.3 安全边界

- 输入器附近持续显示“AI 提供健康信息和就医协助，不替代医生诊断”。
- 出现胸痛、呼吸困难、意识障碍、大量出血等高风险表达时，优先展示急诊提示和 `120` 指引，不用普通建议卡弱化风险。
- 金额、就诊人、医生和时间在最终提交前再次明文确认。
- 颜色不是状态的唯一表达，所有状态配合文字和图标。

## 9. 可访问性

- 结构使用语义化标题、导航、主内容、表单和对话区域。
- 交互元素具备键盘焦点样式和明确的可访问名称。
- 消息流更新使用克制的 `aria-live`，避免逐 token 反复播报。
- 任务面板打开后管理焦点，关闭后将焦点还给触发按钮。
- 文本与背景满足 WCAG AA 对比度；正文不小于移动端可读下限。
- Enter 发送、Shift+Enter 换行；组合输入法未结束时不得误发送。

## 10. 验收标准

- 患者登录后进入模式选择页，两个入口均可用且上次选择只做高亮。
- 新版和传统版可以在不退出登录的情况下双向切换。
- 医生端路由和前端页面不可访问，医生账号不会落入患者页面。
- 新版在 390px 首屏突出 Agent、欢迎语和输入器，不显示传统功能宫格。
- 挂号与缴费任务可在工作台内打开、返回、确认和处理失败状态。
- 对话流式失败、患者上下文部分失败和无数据状态不会阻塞其余功能。
- 360px、390px、768px、1024px 和 1440px 宽度无页面级横向溢出。
- 键盘操作、焦点管理、减少动态效果和医疗免责声明均可验证。
- `npm run lint` 与 `npm run build` 通过。

## 11. 实施顺序

1. 收紧为患者端路由并建立模式选择页。
2. 将传统患者端恢复为可独立进入的完整模式，并加入模式切换。
3. 拆分 Agent 单文件，建立状态 Hook、组件边界和响应式外壳。
4. 实现欢迎态、对话流、上下文摘要和错误状态。
5. 实现通用任务面板及挂号、缴费任务呈现。
6. 完成视觉系统、动效、可访问性和全尺寸验证。
7. 删除医生端前端文件并执行最终回归。
